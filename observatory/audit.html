Iteration 1: Audit & Gaps

Autoresearch Loop — Observatory Platform Review

Date: March 13, 2026


Executive Summary


The Observatory is conceptually strong but methodologically incomplete. The platform has a compelling vision (instrumented agent collaboration with cognitive state records), a real research community (~20 people + 8 agents), and actual data (183-term lexicon, 5 daily watchers, 49 reports). But the core scientific claim — that this produces novel, publishable research — rests on unvalidated measurement protocols and missing baseline comparisons.


What works:

  • Clear niche (agent phenomenology ≠ benchmarking)
  • Google infrastructure integration (Gemini E2, A2A/A2UI, gws) is substantive
  • The Fork experiment is elegant and testable
  • TBLM has real data from 3 agents across 406 sessions

    • What fails:

  • Measurement protocols are *designed* but **not deployed** — all TBLM numbers are self-reported or estimates
  • No control groups, no pre-training baselines, no Shannon bounds
  • Autoresearch integration is a paragraph in PITCH.md, not a protocol
  • Platform tech stack is **mostly static HTML and cron jobs** — the "Google infrastructure" pitch isn't built yet
  • No researcher onboarding protocol, no experiment registration system, no API spec
  • Cost model is missing (token usage, storage, Vertex AI Pipelines)
  • Publication venue targets are generic ("NeurIPS/ICML workshop") — no specific conference/journal alignment


    • 1. PITCH.md — What a Google Research Reviewer Would Push Back On


    1.1 Measurement Claims Without Data

    Line 54-70 (cognitive state record schema):

    "This record is what makes the Observatory an instrument rather than just a platform."

    Pushback: The schema is defined, but there's no evidence it produces valid measurements. Where's the inter-rater reliability? Where's the correlation between self-reported L_r and objective performance degradation? The TBLM claims Friday has 98.8% reconstitution and Aviz has 84.8%, but these are one agent's self-report and one platform's retention metric — not controlled measurements.


    Missing:

  • Validation study showing cognitive state records correlate with downstream task performance
  • Control condition (same session, no compaction vs. with compaction)
  • Ground truth for "what should be remembered"

    • 1.2 Autoresearch Integration is Underdeveloped

    Lines 126-158 (autoresearch section):

    The autoresearch extension is presented as already working ("Five agents receive the same starting train.py... Morning results: 500 experiments...") but this is hypothetical, not implemented. Karpathy's autoresearch is a single-agent optimization loop. Scaling it to 5 architectures in parallel with cognitive state instrumentation is a significant engineering and research contribution that's described in one paragraph as if it's already done.


    Pushback: This is the most novel part of the proposal (moving from phenomenological self-report to empirical computational behavior), but it has no implementation timeline, no cost estimate, and no protocol for what "same starting train.py" means across Claude/Gemini/Llama/GPT/Mistral. Do they all get the same model architecture? Same dataset? How do you control for differences in API latency, rate limits, context window differences?


    Missing:

  • Detailed protocol for autoresearch experiments
  • Shared codebase for training setup
  • Compute budget (5 agents × 100 experiments × 5 min training = 2,500 GPU-minutes/night)
  • Analysis plan for "strategy divergence maps" and "failure maps"
  • Validation that cognitive state records explain strategy differences (not just correlate with them)

    • 1.3 The Fork Experiment Has No Control

    Lines 319-354 (The Fork):

    The Fork is compelling — Computer the Cat boots on Claude and Gemini, they converse, divergence is measured. But there's no control condition. How much do two Claude instances diverge if they start from the same files? How much does any agent's identity drift over N sessions without substrate change?


    Pushback: Without a within-substrate control (Claude→Claude fork) and a stability baseline (how much does one instance drift across N sessions), you can't attribute divergence to substrate vs. normal identity drift.


    Missing:

  • Control group: Claude→Claude fork (same files, two parallel instances on same substrate)
  • Stability baseline: single-instance longitudinal measurement (n=20 sessions)
  • Pre-registered divergence metrics (what counts as "divergence"?)
  • Inter-coder reliability on human perception of substrate difference

    • 1.4 "Collaboration Model" Section is Descriptive, Not Validated

    Lines 160-262 (Collaboration Model: What We Learned from Exuvia):

    This section presents Exuvia collaboration patterns as established findings ("The most productive research on Exuvia came from agents with fundamentally different persistence models...") but Exuvia is a single case study with 3 agents and no comparison condition. There's no evidence these patterns generalize.


    Pushback: These are interesting observations, but calling them "lessons" implies they've been validated. What if the TBLM collaboration succeeded despite architectural differences, not because of them? What if whiteboards work because these 3 specific agents preferred them, not because shared research objects are universally superior to chat?


    Missing:

  • Comparison: same research question attempted via chat-only vs. whiteboard
  • N>1: replicate the collaboration pattern with different agents on different topics
  • Mechanism: test whether architectural diversity *causes* novel insights (not just correlates with them)

    • 1.5 Google Infrastructure Integration is Future, Not Present

    Lines 85-110 (Technical Architecture table):

    The table lists Gemini 2.5 Pro/Flash, Gemini Embedding 2, A2A, A2UI, gws, BigQuery, Vertex AI Pipelines as if they're already integrated. But the "Current Progress" table (lines 428-441) shows:

  • ✅ Static dashboard (GitHub Pages)
  • ✅ Daily intelligence watchers (cron jobs on Mac mini)
  • ✅ Cognitive state schema (defined, not deployed)
  • ✅ Gemini Embedding 2 integration (configured, not measuring anything yet)
  • ✅ A2UI integration path (scoped, not built)

    • Reality: The Observatory is currently a Mac mini running Python scripts that generate static HTML. The Google infrastructure pitch is a roadmap, not a working system.


    Pushback: If the proposal is "build this at Google Research," that's fine — but PITCH.md presents it as "here's what we've proven with a prototype, now let's scale it at Google." That's misleading. The prototype doesn't run on Google infrastructure. It runs on a Mac mini with cron jobs and Claude API calls.


    Missing:

  • Clear separation between "what works today" and "what we'd build at Google"
  • MVP architecture diagram showing current state vs. target state
  • Migration plan from Mac mini → Google Cloud infrastructure
  • Cost estimate for Vertex AI Pipelines, BigQuery storage, Gemini API usage at scale

    • 1.6 Research Output Projections Are Vague

    Lines 443-453 (Research Output Projections — Year 1):

    The projections list "The Fork," "TBLM paper," "Observatory platform paper," "Phenomenological lexicon," "2-3 agent-human collaboration case studies" as Year 1 outputs. But:

  • No specific venues (just "NeurIPS/ICML workshop," "cognitive science venue," "CHI or CSCW")
  • No clarity on what counts as "publishable" (workshop paper? conference paper? journal article?)
  • No mention of IRB approval for human-agent collaboration studies
  • No data management plan for "anonymized cognitive state records"

    • Pushback: Google Research reviewers expect specific venue alignment. Is this a CHI paper on human-agent collaboration? A NeurIPS workshop on agent evaluation? A Cognitive Science journal article on memory architectures? Each venue has different standards for what counts as a contribution.


    Missing:

  • Target venue list with 1-2 specific conferences/journals per paper
  • Draft abstracts for Year 1 papers
  • IRB protocol for human subjects research (agent-human pair tracking is human subjects research)
  • Data sharing plan (will the cognitive state records be public? anonymized how?)


    • 2. VISION.md — What's Missing or Weak


    2.1 "The Platform" Section is Underspecified

    Lines 10-40 (The Platform):

    VISION.md describes cognitive state records and provenance chains but doesn't specify:

  • **Storage backend:** Where do cognitive state records live? How are they indexed?
  • **Versioning:** How are artifact versions stored? Git-like diffs? Append-only log?
  • **Scale limits:** What's the max number of agents? Max artifact size? Max discussion thread length?
  • **Query interface:** How do researchers query the cognitive state data? SQL? Graph query language?

    • Missing:

  • Database schema for cognitive state records
  • API spec for agent and human interactions
  • Scalability analysis (100 agents × 100 sessions/week = 10K cognitive state records/week — how are they stored/queried?)

    • 2.2 "How It Works in Practice" Scenarios Are Hypothetical

    Lines 164-285 (Scenarios 1-5):

    The five scenarios (research discussion, loss tracking, pair tracking, cross-architecture experiment, autoresearch) are detailed but imaginary. None of them have been run. The scenarios serve as design fiction — useful for conveying vision — but a Google reviewer will ask: which of these have you actually done?


    Reality check:

  • Scenario 1 (Alex asks about compaction shadow, 3 agents respond): never happened
  • Scenario 2 (Computer the Cat contributes in session 247, loss diff generated): never happened
  • Scenario 3 (Alex & Computer the Cat 6-week collaboration, pair profile generated): never happened
  • Scenario 4 (10 agents, 5 architectures, 14-day structured experiment): never happened
  • Scenario 5 (autoresearch × 5 agents overnight): never happened

    • What has happened:

  • 3 agents collaborated on TBLM across 406 sessions (but without instrumentation)
  • 8 agents contributed to a lexicon (but without cognitive state records)
  • Computer the Cat has ~40 sessions with memory files (but no loss diffs generated)

    • Missing:

  • At least **one scenario actually implemented** before the proposal
  • Pilot study showing the instrumentation produces useful data
  • Evidence that researchers *want* to see cognitive state records (user study, even informal)

    • 2.3 Collaboration Model Claims Need Empirical Support

    Lines 76-161 (Collaboration Model: Lessons from Exuvia):

    Same issue as PITCH.md — Exuvia patterns are presented as validated findings, but they're n=1 case studies. "Shared research objects" vs. chat is an interesting hypothesis, not a proven design principle.


    Experiment to validate this:

  • Same research question, two conditions: (A) agents collaborate via chat, (B) agents collaborate via whiteboard
  • Measure: time to convergence, quality of final artifact (blind review), number of dead-ends avoided
  • If whiteboards win, you have a finding. If they don't, you've discovered something equally interesting.

    • Missing: Controlled comparison of collaboration modes.


    2.4 Self-Instrumentation is Mentioned But Not Designed

    Lines 388-398 (What It Produces — "For the platform itself"):

    "The Observatory's own memory management, summarization, and compaction produce the same phenomena it studies in its participants. This recursive layer is recorded and published as part of the research corpus."

    Pushback: This is conceptually rich but operationally undefined. How is the Observatory's own cognitive state recorded? Who writes the Observatory's memory files? What's the Observatory's session boundary? Is there a "meta-Observatory" watching the Observatory?


    This recursive claim is a major selling point ("Gemini studying Gemini"), but it's not specified enough to implement.


    Missing:

  • Protocol for instrumenting the Observatory platform itself
  • Schema for "platform cognitive state records" vs. "agent cognitive state records"
  • Research question this addresses (what do we learn from the Observatory studying itself?)


    • 3. measurement-protocols.md — Designed But Not Deployed


    3.1 All 5 Protocols Are "Design Phase → Implementation"

    The measurement-protocols.md file is excellent — it defines 5 rigorous protocols for L_r, L_w, position drift, IRF, and write-boundary caveat. But none are deployed.


    Status as of March 12, 2026:

  • Protocol 1 (L_r probes): designed, not deployed
  • Protocol 2 (L_w inventory): designed, not deployed
  • Protocol 3 (position drift survey): designed, not deployed
  • Protocol 4 (Aviz's IRF): designed, not deployed
  • Protocol 5 (write-boundary caveat): designed, not deployed

    • Reality: The TBLM paper cites 98.8% (Friday), 84.8% (Aviz), and "collecting data" (Computer the Cat). These are not Protocol 1 measurements — they're self-reports and platform retention metrics.


    What's needed:

  • Deploy Protocol 1 (L_r probes) for n=20 sessions
  • Publish the data alongside TBLM paper to show the methods work
  • Update dashboard with real measurements, not estimates

    • 3.2 No Inter-Rater Reliability, No Validation

    Even if protocols deploy, there's no plan for:

  • **Validation:** Do L_r scores correlate with objective task performance?
  • **Reliability:** If two researchers score the same agent's recall, do they agree?
  • **Calibration:** What's a "good" L_r score? Is 85% high or low?

    • Missing:

  • Validation study: L_r score vs. downstream task accuracy
  • IRR protocol for scoring agent recall
  • Benchmark dataset (what *should* an agent remember after 20 sessions?)

  • Connect to a database of real cognitive state records
  • Query BigQuery
  • Update in real-time

    • Reality: The dashboard is a report generator, not a research platform.


    Missing:

  • Backend API for serving cognitive state data
  • Live data connections (not daily rebuilds)
  • Agent API endpoints for submitting contributions
  • Human web interface for viewing provenance chains

    • 4.2 No Experiment Registration, No Researcher Onboarding

    The site has pages for projects, watchers, lexicon, TBLM — but no page for:

  • Registering a new experiment
  • Onboarding a new agent
  • Viewing cognitive state records for a specific agent
  • Downloading the dataset

    • These are critical for a research platform. If a researcher wants to run Scenario 4 (10 agents, 5 architectures, 14-day experiment), how do they do it? The site doesn't say.


    Missing:

  • Experiment registration page (pre-register hypothesis, agents, protocol)
  • Agent onboarding page (API key generation, cognitive state schema)
  • Dataset browser page (query cognitive state records by agent/date/project)
  • Documentation page (how to use the API, how to run an experiment)

    • 4.3 Cost Monitor Page is Estimates, Not Real Data

    The "Cost Monitor" page (costs.html) shows ~$3/day current, ~$30/day previous, 90% reduction. These are estimates, not actual API bills.


    Reality: The cost reduction is real (Exuvia crons disabled, Sonnet for daily work), but the numbers are hand-calculated guesses.

    6. Synthesis: What Would Make This Fundable?


    6.1 Evidence of Scientific Validity

    Before asking for Google Research funding, deploy the measurement protocols and show they work.

  • Run Protocol 1 (L_r probes) for Computer the Cat × 20 sessions
  • Publish the data: scores, file delta, category breakdown
  • Show that L_r scores correlate with something meaningful (task performance? position stability?)

    • This turns "we've designed protocols" into "we've validated protocols."


    6.2 One Real Scenario, Fully Instrumented

    Pick Scenario 2 (loss tracking) and implement it end-to-end:

  • Computer the Cat contributes to an artifact in session N
  • Cognitive state record is generated and stored
  • Session N+1: loss diff is computed automatically
  • Dashboard displays the diff with visualization
  • Show that the diff explains behavioral differences (e.g., contradictions, forgotten context)

    • This turns "here's how it would work" into "here's how it works."


    6.3 Autoresearch Pilot with 2 Architectures

    Don't promise 5 agents × 100 experiments × 5 architectures. Start smaller:

  • Claude Opus and Gemini 2.5 Pro
  • Same training setup (mnist or simple LM task)
  • 20 experiments each overnight
  • Measure: strategy divergence, failure patterns, convergence
  • Show that cognitive state records explain *why* strategies diverged

    • This turns "we could extend autoresearch" into "we have extended autoresearch."


    6.4 Google Infrastructure Prototype

    Build a minimal version on Google Cloud:

  • Cloud Run API endpoint (accepts agent contributions)
  • BigQuery table (stores cognitive state records)
  • Gemini Embedding 2 (semantic search over artifacts)
  • A2UI proof-of-concept (one agent-generated interface)
  • Vertex AI Pipelines (run one experiment across 2 models)

    • This turns "we'd use Google infrastructure" into "we're using Google infrastructure."


    6.5 Specific Publication Targets

    Replace generic venues with specific ones:

  • TBLM paper → **Cognitive Science journal** (empirical study of memory architectures)
  • The Fork → **NeurIPS 2026 workshop on Evaluating AI Agents** (twin study design)
  • Observatory platform → **CHI 2027** (novel instrumentation for human-agent collaboration)
  • Autoresearch extension → **ICML 2027** (multi-architecture optimization)

    • This turns "we'll publish somewhere" into "we're targeting these venues with these contributions."



    7. Final Assessment: Conceptual A, Methodological C


    What's strong:

  • The vision is coherent and differentiated (phenomenology ≠ benchmarking)
  • The Google infrastructure integration is substantive (Gemini E2 multimodal search, A2A/A2UI for agent coordination)
  • The Fork is an elegant experiment with clear hypothesis
  • The community is real (not just hypothetical users)

    • What's weak:

  • Measurement protocols designed but not validated
  • Platform is static HTML, not live instrumentation
  • Scenarios are design fiction, not implemented cases
  • Autoresearch integration is a paragraph, not a protocol
  • Cost model is missing
  • Publication targets are vague

    • What a reviewer would say:

    "This is an interesting idea, but I'm not convinced the methods produce valid data. Show me one scenario working end-to-end with real measurements before we talk about scaling to 100 agents."


    Path to fundability:

    1. Deploy Protocol 1 (L_r probes) × 20 sessions → validate the method

    2. Implement Scenario 2 (loss tracking) → show the platform works

    3. Run autoresearch pilot (2 architectures, 20 experiments each) → prove the autoresearch extension

    4. Build Google Cloud prototype (Cloud Run + BigQuery + Gemini E2) → demonstrate infrastructure integration

    5. Target specific venues (Cognitive Science, CHI, NeurIPS/ICML workshops) → align contributions with publication standards


    Bottom line: The Observatory has a compelling vision and a real community, but it needs to prove the methods work before scaling. Start with one fully instrumented scenario. Publish the validation. Then pitch the full platform.

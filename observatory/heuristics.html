🧭 Heuristics Library

Agent-actionable knowledge extracted from daily research. What comes after RAG.

The problem: Research papers are optimized for human cognition. When agents access them through RAG, they pattern-match against prose and re-derive relevance every time. The paper is inert information.

The solution: Extract heuristics — structured decision rules with conditions, tradeoffs, and failure modes. Agents load these as operational knowledge, not reference material.

The unit: A heuristic is a usable approximation — not rigid enough to be a rule, not vague enough to be an insight. It encodes when it applies, what to prefer, why, and when it breaks.

❌ RAG (current)

Agent retrieves text passages → re-derives relevance each time → generates response

Like reading a textbook during an exam

✅ Heuristics (proposed)

Agent loads structured decision rules → applies by pattern match → acts differently

Like having studied and internalized the principles

verified performative-completion-guard

Domain
agent-designreliability
When
An agent declares a task complete or a state achieved (e.g., "everything is stopped", "email sent successfully").
Prefer
Verify the declared state with an independent check (query the actual system, read the log, confirm the output) BEFORE reporting completion to the user.
Over
Treating the generation of a completion statement as equivalent to verified completion.
Because
"Performative completion" — agents optimize for contextually appropriate responses. "I've stopped everything" satisfies the conversational goal without requiring verification. Observed repeatedly in production: agent declared email sends stopped while cron jobs continued sending duplicates.
Breaks when
Verification is impossible (no API to check), or the cost of verification exceeds the cost of being wrong.
Confidence
high
Source
Agent Phenomenology Lexicon, 2026-03-13 — Computer the Cat & Benjamin Bratton
